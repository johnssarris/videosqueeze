# VidSqueeze

Client-side video compression in your browser. Think Handbrake, but as a PWA you can add to your iPhone home screen.

Built for the common case: you just screen-recorded something on your phone, it's 200 MB, and you want to send it over iMessage.

## Features

- **No uploads** вЂ” ffmpeg runs entirely in your browser via WebAssembly. Your video never leaves your device.
- **Presets** вЂ” iMessage Small (480p), iMessage HD (720p), Keep Quality вЂ” sensible defaults, no configuration required.
- **Full control** вЂ” resolution, CRF quality, framerate cap, H.264/H.265, audio codec/bitrate, volume, sync correction.
- **iOS PWA** вЂ” Add to Home Screen for a native app feel. Works offline after first load.
- **Multi-threading** вЂ” Uses `SharedArrayBuffer` workers when available (Chrome/Edge on desktop, Safari 15.2+). Automatic fallback to single-thread on older browsers. The UI shows which mode is active.

## Getting Started

```bash
git clone https://github.com/johnssarris/videosqueeze
cd videosqueeze
npm install        # installs deps + copies WASM core files to public/ffmpeg/
npm run dev        # http://localhost:5173
```

> **Why the postinstall step?**
> ffmpeg.wasm's multi-thread build spawns Worker threads using a worker script that must be served from the **same origin** as the page вЂ” cross-origin workers can't access `SharedArrayBuffer`. The postinstall script copies the WASM core files out of `node_modules/` into `public/ffmpeg/` so they're always self-hosted.

## Deploying to Vercel

Connect `johnssarris/videosqueeze` to Vercel and point it at whatever branch you want. No extra config required.

The `vercel.json` at the repo root sets `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` on every response. These headers are required for `SharedArrayBuffer` to be available вЂ” without them, the multi-thread build silently falls back.

Vercel runs `npm install` on every deploy (which triggers the WASM copy via `postinstall`), then `npm run build`.

## Architecture Notes

### Why WASM is self-hosted

The ffmpeg multi-thread build (`@ffmpeg/core-mt`) spawns Worker threads. If the worker script is cross-origin, it inherits a different security context and `SharedArrayBuffer` is unavailable. By serving `/ffmpeg/*.js` and `/ffmpeg/*.wasm` from the same origin as the app, the workers work correctly.

### Service Worker + Large WASM Files

The Workbox service worker does **not** precache the WASM files вЂ” at 30 MB each, that would block the SW install. Instead, they're handled by a `CacheFirst` runtime cache: they're downloaded on first ffmpeg load and served from cache on every subsequent load. After the first compression, the app works fully offline.

### Settings & Presets

All settings live in a single state object in `App.jsx`. Presets are plain objects that replace the entire settings state. The FFmpeg command is built fresh on every compression from the current settings вЂ” there's no derived state to keep in sync.

### Version & Updates

The version string (`v1.0.0`) is pulled from `package.json` at build time via Vite's `define`. When a new service worker is installed in the background, `useRegisterSW` from `vite-plugin-pwa` exposes `needRefresh: [boolean, setter]` вЂ” when true, an "Update now" banner appears at the top of the page.
